# 预言机集成审计要点（Chainlink + RedStone）

这份文档是审计 `SolvencyFacetProd` 的 RedStone/Chainlink 价格消费路径时沉淀下来的通用知识点。

**定位**：记录的是**可迁移到其他项目的知识**，本仓库的代码只作为案例出现。具体某一条在本仓库是不是真漏洞、影响多大，不在这份文档的范围内。

**知识边界**：文中 RedStone 部分的所有结论都来自实际阅读 `@redstone-finance/evm-connector@0.9.0` 的源码（见[附录](#附录怎么读一个没装的依赖的源码)）。Chainlink 部分的通用机制描述来自公开文档和历史事件，但**具体某个 feed 的 heartbeat/deviation 数值必须自己去查**，不要信任何二手数字（包括这份文档里的举例）。

> **本篇很多论断已在 DeltaPrime 的真实部署上用链上数据验证过**，验证方法和实测数字见[第五章](#五实测验证方法与本仓库的真实数据)。读通用知识时可以先跳过第五章，但如果你怀疑某条"是不是真的"，那一章给的是可复现的证据。

## 一、Chainlink 篇

### 1.1 最反直觉的一点：feed 不是"实时"的，是"事件驱动"的

这是我之前完全没有概念、也是最容易在审计里漏掉的一点。

**Chainlink 的价格不是每个区块更新，甚至不是定期更新的。** 每个 feed 有两个独立的触发条件，满足任一个才会推送新价格上链：

| 触发条件 | 含义 |
| --- | --- |
| **Deviation threshold**（偏离阈值） | 价格相对上次推送偏离超过 X% 就立刻更新 |
| **Heartbeat**（心跳） | 距上次推送超过 Y 时间就强制更新一次，哪怕价格纹丝不动 |

也就是说：**heartbeat 是"最坏情况下多久更新一次"的上界，不是"通常多久更新一次"。**

关键推论：**在行情平静时，一个完全健康、完全符合规格的 feed，可以整整一个 heartbeat 周期不更新。** `updatedAt` 是几个小时前，这不是异常，这是设计。

#### 数值有多大？必须逐 feed 查

这两个参数**每个 feed、每条链都不一样**，跨度极大。粗略的感觉：

- 主流资产 + 主网（如 ETH/USD）：偏离阈值小、heartbeat 短（小时级）。
- 长尾资产 / L2 / 备选 L1（Arbitrum、Avalanche 等）：偏离阈值大、**heartbeat 常见 24 小时**。
- 稳定币 feed：偏离阈值往往设得很小（因为本来就不该动），heartbeat 反而很长。

> ⚠️ **不要记具体数字，要记查询方法。** 去 [data.chain.link](https://data.chain.link) 选中链和 feed，页面上直接标着 deviation threshold 和 heartbeat。或者链上读 aggregator。同一个 `ETH/USD` 在不同链上参数可以差一个数量级。

#### 为什么这直接决定了 staleness 检查怎么写

审计里最常见的写法：

```solidity
uint256 private constant CHAINLINK_MAX_DELAY = 6 hours;   // 全局常量
...
if (block.timestamp - updatedAt > CHAINLINK_MAX_DELAY) revert ChainlinkStaleData();
```

**一个全局常量伺候所有 feed，必然在两个方向上同时出错**：

- 常量 **小于** 某个 feed 的 heartbeat（例：6h 常量 vs 24h heartbeat）
  → 该 feed **正常工作**时就会被判定为 stale → **误报**。
- 常量 **大于** 某个 feed 的 heartbeat（例：6h 常量 vs 1h heartbeat）
  → 该 feed 已经挂了 5 小时，检查却放行 → **漏报**，正是这个检查存在的意义所在。

**正确做法：staleness 阈值必须 per-feed 配置**，和 feed 地址存在一起（比如 TokenManager 里），取值 = 该 feed 的 heartbeat + 一点缓冲。

#### 误报的代价可能比漏报更大 ⚠️

这一点很违反直觉，值得单独强调。

Staleness 检查失败时是 `revert`。如果这个检查位于**偿付能力计算路径**上（`isSolvent()` / `getHealthRatio()`），而这条路径又被 `remainsSolvent` 之类的修饰器挂在**每一个用户操作**上，那么一次误报的后果是：

```
getChainlinkPrice() revert
  → getPrices...() revert
    → isSolvent() revert
      → 所有用户操作 revert  +  清算也 revert
```

**协议被整个砖掉**：用户不能减仓自救，清算人不能清算，而坏账在这段时间里继续累积。

所以别把 "revert" 想当然地当成 "fail-safe"。**要问的是：这个 revert 冒泡上去之后，谁被挡住了？** 如果清算路径也被挡住，那这是 fail-**closed-and-broken**，不是安全。

（对照：本仓库 [SolvencyFacetProd.sol:54](../contracts/facets/SolvencyFacetProd.sol#L54) 的常量是 `6 hours`，而紧挨着的注释写的是 `///@dev updated to 25 hours` —— 代码和注释互相矛盾，是这类问题的典型信号。）

### 1.2 `minAnswer` / `maxAnswer` 熔断：`answer > 0` 拦不住的那类错误

Chainlink 的 aggregator 内部有价格上下界（`minAnswer` / `maxAnswer`）。**当真实价格穿透边界时，feed 不会报错，而是老老实实返回那个边界值** —— 一个格式完全合法、非零、不 stale，但和现实完全脱节的数字。

#### 历史事件：Venus / LUNA（2022 年 5 月）

LUNA 崩盘到接近归零时，对应 feed 的 `minAnswer` 卡在 $0.107 左右。Venus Protocol 于是继续按 $0.107 给 LUNA 抵押品估值，而它的真实市价已经是这个数的零头。攻击者用近乎白送的 LUNA 换到了被高估几十倍的抵押品价值，借空了其他资产。

**所有常规检查在这里全部通过**：`answer > 0` ✅、不 stale ✅、`roundId` 正常 ✅。这就是为什么这类检查必须单独做。

#### 怎么检查

```solidity
// 注意：minAnswer/maxAnswer 在 aggregator 上，不在 proxy 上
address agg = AggregatorV3Interface(feed).aggregator();
int192 minAnswer = IOffchainAggregator(agg).minAnswer();
int192 maxAnswer = IOffchainAggregator(agg).maxAnswer();
if (answer <= minAnswer || answer >= maxAnswer) revert PriceAtCircuitBreakerBound();
```

几个坑：

- **必须走 `.aggregator()`**，`AggregatorV3Interface`（proxy）不暴露这两个值。
- **aggregator 地址会变**（Chainlink 升级 feed 时换 aggregator，proxy 地址不变）。缓存 aggregator 地址会踩雷，要每次读或者留升级通道。
- **不是所有 feed 都有意义的边界**。事件之后 Chainlink 在很多 feed 上把边界放到了 `type(int192).min/max` 这种实际等于禁用的值。有边界的 feed 才需要这个检查 —— 但**别假设**某个 feed 没有，去查。

### 1.3 `answeredInRound` 已经废弃了

这个检查在老教程里到处都是：

```solidity
if (answeredInRound < roundId) revert ChainlinkStaleRound();
```

**在现代 OCR / OCR2 feed 上，`answeredInRound` 恒等于 `roundId`，这个条件永远为假。** 它来自 OCR 之前的旧聚合器架构，那时一轮可能没凑够答案而沿用上一轮的值。

危害不在于它做错了什么（它什么都没做），而在于**它提供虚假的安全感**：审计者扫过去看到"哦，陈旧轮次处理了"，就不再追问真正的 staleness 保护够不够。

> 📌 **通用教训**：一个恒为假的检查比没有检查更糟。没有检查会被 review 抓出来，假检查会让 review 直接跳过这一块。

Chainlink 官方现在推荐直接用 `updatedAt` 做新鲜度判断，`answeredInRound` 已从推荐用法中移除。

### 1.4 decimals 归一化的截断陷阱

跨 feed 归一化到统一精度时的常见写法：

```solidity
} else if (feedDecimals > 8) {
    price = uint256(answer) / (10 ** (feedDecimals - 8));   // 截断除法
}
```

`answer > 0` 检查是在归一化**之前**做的。一个 18 位小数的 feed 上，低价代币归一化到 8 位后可能被舍成 **0** —— 一个通过了所有校验的零价格。

两条教训：

1. **校验要放在最终使用的那个值上，不是中间值上。** 归一化之后要重新检查 `price != 0`。
2. **零价格往往不会 revert，而是安静地传播下去。** 更糟的是，下游代码常有 `if (price == 0) return xxx;` 这类"宽容"分支（本仓库 [SolvencyFacetProd.sol:615](../contracts/facets/SolvencyFacetProd.sol#L615) 的 `_getSolvencyBalance` 就有：价格为 0 时直接返回原始余额，**跳过手续费扣减**）。零价格 + 宽容分支 = 静默跳过一整段逻辑。

## 二、RedStone 篇

### 2.1 Pull 模型：价格在 calldata 里，不在链上

和 Chainlink（push 模型，价格已经躺在链上，你只管读）最根本的区别：

**RedStone 的价格由调用者自己把签名 payload 追加在 calldata 末尾，合约在 `msg.data` 尾部现场解析、验签、聚合。**

```
[ 正常 ABI 编码的函数参数 ][ RedStone payload ][ marker ]
                              ↑ 调用者自己拼上去的
```

推论，每一条都影响审计视角：

- **"谁调用"和"用什么价格"是同一个人。** 攻击者可以在他选定的时刻、用他选定的（但必须是签过名的）价格调用。
- **价格新鲜度由调用者的选择决定**，只受时间戳窗口约束 —— 攻击者会挑窗口内对自己最有利的那一份。
- **合约不能主动"读"价格**，只能在有 payload 的调用里被动解析。所以任何路径上只要 calldata 没带 payload，解析就 revert。
- SDK 会自动包装调用来附加 payload，所以**在测试里这层是隐形的** —— 这也是它容易被审计漏掉的原因。

### 2.2 ⭐ 所有安全钩子都是 `virtual` 的 —— 攻击面不在你读的文件里

**这是审计 RedStone 集成时最重要的一条。**

`RedstoneConsumerBase` 把全部安全参数做成了 `virtual` 函数，等着子类 override：

| 钩子 | 默认行为 | 被 override 的后果 |
| --- | --- | --- |
| `getUniqueSignersThreshold()` | 由 data-service 基类定，如 `PrimaryProdDataServiceConsumerBase` = **3** | 改成 1 → 单点签名者即可操纵价格 |
| `getAuthorisedSignerIndex(addr)` | 硬编码授权签名者地址集，未授权则 revert | 加入攻击者地址 → 任意价格 |
| `validateTimestamp(ts)` | `RedstoneDefaultsLib`：过去 **3 分钟** / 未来 **1 分钟** | 放宽 → 陈旧价格重放 |
| `aggregateValues(values)` | 中位数（`pickMedian`） | 改成取最大/最小 → 单签名者即可拉偏 |

**所以"这个 facet 的价格安全吗"这个问题，看这个 facet 的源码是答不出来的。** 必须 grep 整个继承树：

```bash
grep -rn "getUniqueSignersThreshold\|getAuthorisedSignerIndex\|validateTimestamp\|aggregateValues" \
     --include=*.sol contracts/
```

然后逐个确认每个 override 的位置。**重点是把 `mock/` 目录里的 override 和生产合约的继承链严格分开** —— 测试用的 mock 几乎总是把门限降到 1、把签名者换成本地测试地址、把 `validateTimestamp` 变成空函数。一旦某个生产合约意外继承了 mock 基类，或者部署脚本把 mock 部署上去了，价格校验就等于**完全不设防**。

> 📌 **通用教训**：任何库把安全参数设计成 `virtual`，"有没有人 override"就是第一位的审计问题，优先级高于读主逻辑。同类模式：OpenZeppelin 的 `_beforeTokenTransfer`、`_authorizeUpgrade`。

### 2.3 Diamond / 代理场景：payload 怎么跨调用传递

这是 RedStone + Diamond 组合的必考题。

**问题**：价格来自 `msg.data` 尾部。A facet 调 B facet 时 calldata 是 `abi.encodeWithSelector(...)` 重新编码的 —— **payload 丢了**，B 里的解析必然 revert。

**解法**：RedStone 官方的 `ProxyConnector`：

```solidity
proxyDelegateCalldata(facetAddress, abi.encodeWithSelector(sel, args))
```

它内部 `_prepareMessage()` 做的事：`calldatacopy` 当前 `msg.data` 的尾部 payload，拼到新编码的 calldata 后面，再 delegatecall。三个变体：

| 函数 | 调用方式 | 用途 |
| --- | --- | --- |
| `proxyCalldata` | `call` | 跨合约，可转 value |
| `proxyDelegateCalldata` | `delegatecall` | Diamond facet 之间（保留 storage 上下文） |
| `proxyCalldataView` | `staticcall` | view 路径 |

**审计要点**：找所有"重新编码 calldata"的地方，确认它们都走了 `ProxyConnector`。裸的 `address(this).call(abi.encodeWithSelector(...))` 是危险信号 —— 目标函数如果消费 RedStone，会因为拿不到 payload 而 revert。

而**如果这个裸调用的失败被 `if (success)` 静默吞掉了，那更糟**：不是 revert，而是这个位置的值被当成 0 处理。本仓库 [SolvencyFacetProd.sol:669](../contracts/facets/SolvencyFacetProd.sol#L669) 的 `_getTWVStakedPositions` 就是这个形状（`staticcall` + `if (success)`，失败则该仓位静默计为 0）。这里方向恰好是保守的（低估自己的资产），但这个模式本身值得警惕。

### 2.4 `getOracleNumericValues*` 系列 API 的区别

用错不会出安全问题，但会白烧 gas，也说明作者没吃透这套 API：

| API | 是否验时间戳 | 是否允许重复 feed id | 备注 |
| --- | --- | --- | --- |
| `getOracleNumericValueFromTxMsg(id)` | ✅ | — | 单个 |
| `getOracleNumericValuesFromTxMsg(ids)` | ✅ | ❌ **要求已去重** | 常规批量 |
| `getOracleNumericValuesWithDuplicatesFromTxMsg(ids)` | ✅ | ✅ | **内部自己再去重一遍**（两层 O(n²) 循环） |
| `getOracleNumericValuesAndTimestampFromTxMsg(ids)` | ❌ **不验！** | ❌ | ⚠️ 返回时间戳让你自己验，忘了验 = 陈旧价格随便用 |

两个坑：

1. **自己已经去重了就别再调 `WithDuplicates` 版本** —— 白白多跑一遍 O(n²)。（本仓库 [SolvencyFacetProd.sol:128](../contracts/facets/SolvencyFacetProd.sol#L128) 正是这样：Pass 2 手工去重出 `globalUniqueSymbols`，Pass 3 又调了 `WithDuplicates` 版本。）
2. **`...AndTimestampFromTxMsg` 是唯一不自动验时间戳的**，函数名完全看不出来这个区别，只有注释里写了。用它就必须自己调 `validateTimestamp`。

### 2.5 空 feed 列表 → 门限检查被完全跳过

一个值得记住的边界情况，读源码才能发现：

```solidity
// RedstoneConsumerBase._getAggregatedValues
for (uint256 i = 0; i < valuesForDataFeeds.length; ) {
    if (uniqueSignerCountForDataFeedIds[i] < uniqueSignersThreshold) {
        revert InsufficientNumberOfUniqueSigners(...);   // ← 循环体内
    }
    ...
}
```

**门限检查在循环体内。`dataFeedIds` 为空 → 循环零次 → 一次检查都不做，直接返回空数组。**

在本仓库里这不可利用（返回空数组，没有价格被使用）。但这是个**通用陷阱模式**：如果某段代码的逻辑是"取 N 个价格 → 校验 → 用"，而 N 可以被推成 0，那么"校验"这一步就凭空消失了。**要单独问：N=0 时会发生什么？**

顺带一个相关观察：`_securelyExtractOracleValuesAndTimestampFromTxMsg` 里**没有** `dataPackagesCount == 0` 的显式检查（对比 `extractTimestampsAndAssertAllAreEqual` 里就有 `RedstonePayloadMustHaveAtLeastOneDataPackage`）。它靠的是"0 个包 → `dataPackagesTimestamp` 保持 0 → `validateTimestamp(0)` → `TimestampIsTooOld` revert"来兜底。**结论是安全的，但靠的是间接的二阶效应，不是显式断言** —— 这种"碰巧安全"的地方在库升级时最容易破。

### 2.6 门限 3-of-5 的隐性成本：payload 体积与重复解析

`getUniqueSignersThreshold() = 3` 意味着**每个 feed 至少要 3 个独立签名的 data package**。一个多资产账户的 payload 可能有几十个包，每个包都要做一次 `ecrecover`。

**所以"再取一次价格"这个动作的代价是：重新解析整个 payload + 对每个包重新 `ecrecover`。**

这带来一个具体的反模式 —— **在循环里回退取价**：

```solidity
// 循环体内
function _findPriceForToken(...) internal view returns (uint256) {
    for (...) { if (prices[i].asset == symbol) return prices[i].price; }  // 先找缓存
    // 缓存里没有 → 回退，重新解析整个 payload
    return getPricesFromRedstoneAndChainlink(single)[0];
}
```

安全性上没问题（没签名就 revert，不会 fail-open）。但如果它在循环里被调用，**gas 是二次方增长的**：N 个未命中缓存的 token × 全量 payload 重解析。

**攻击价值在于 DoS 清算**：只要能构造出让清算交易的价格解析次数变多的仓位（比如持有多个市场、且把 underlying 挡在 `ownedAssets` 之外），就能把清算的 gas 推高到无利可图、甚至撞上区块 gas 上限 —— **清算不了 = 坏账**。

> 📌 **通用教训**：在 pull oracle 里，**"取价格"是重操作，不是 getter**。任何在循环里取价的代码都要按 DoS 视角审。正确做法是进循环前一次性批量取全。
> （本仓库案例：[SolvencyFacetProd.sol:643-653](../contracts/facets/SolvencyFacetProd.sol#L643-L653)。）

## 三、⭐ 混合预言机篇：新鲜度不对等

**如果这份文档只带走一条，是这条。** 它是架构层面的，grep 不出来，只有把两条路径的参数并排放才看得见。

### 3.1 问题

同一个合约里，RedStone 和 Chainlink 的时间窗差了两个数量级：

| 来源 | 最大陈旧度 | 来自哪 |
| --- | --- | --- |
| RedStone | **3 分钟** | `RedstoneDefaultsLib.DEFAULT_MAX_DATA_TIMESTAMP_DELAY_SECONDS` |
| Chainlink | **数小时**（等于你设的 staleness 常量） | 项目自己的常量 |

**同一次 `getHealthRatio()` 调用，会把一个 3 分钟新鲜的价格和一个可能几小时陈旧的价格放进同一个除法里。**

两个价格各自都"合法"，但它们描述的**不是同一个时刻的世界**。健康度算出来是个跨越几小时的混合体。

### 3.2 为什么路由规则会让偏差变成单向的

如果新鲜/陈旧在分子分母上随机分布，那还只是噪声。但常见的路由规则会让它**系统性偏向一边**。

本仓库的规则（[SolvencyFacetProd.sol:77-105](../contracts/facets/SolvencyFacetProd.sol#L77-L105)）：**可借资产（债务）强制走 RedStone；其他资产配了 Chainlink feed 就走 Chainlink。**

代入公式：

```
健康度 = 抵押品价值（可能是 Chainlink，几小时陈旧）
        ────────────────────────────────────────
         债务价值（强制 RedStone，3 分钟新鲜）
```

**分母永远新鲜，分子可能陈旧。偏差是单向的、可预测的、可选择时机的。**

于是攻击者的操作是确定性的，不需要任何"机会"：

1. 盯一个走 Chainlink 的抵押品，等它的链上价格显著**高于**真实市价 —— 注意，**这不需要 feed 出故障**，只要偏离还没到 deviation threshold、heartbeat 也还没到（见 [1.1](#11-最反直觉的一点feed-不是实时的是事件驱动的)），feed 就是这么设计的。
2. 此时健康度 = 虚高的抵押品 / 真实的债务 → **虚高**。
3. 按几小时前的价格超额借款。

反方向同样成立：价格上涨时，走 Chainlink 的抵押品还停在旧的低价，用户会被**按陈旧价格错杀清算**。

### 3.3 为什么保守化手段救不了

`Math.min(...)`、debt coverage 折扣这类"取保守值"的手段，在这里**没用**。因为偏差发生在价格**进入公式之前** —— 你对一个错误的输入取 min，得到的还是错误的输出。

### 3.4 审计方法

**把每条价格路径的时间窗做成一张表，并排放。** 只要行与行之间的数字不在一个量级，就去问：

1. **这个差值能落在公式的哪一侧？** 分子和分母的新鲜度是否对称？
2. **路由规则是否让偏差单向化？**（"债务必须用 X" 这类规则是最强的信号）
3. **谁能选择调用时机？** 在 DeFi 里答案基本永远是"攻击者"。

**根治方向**：混合预言机时，**所有路径的新鲜度窗口应当对齐到最严格的那一个**。做不到（Chainlink 的 heartbeat 是硬约束，你没法让它更新更快）的话，就得承认这个资产不适合和 pull oracle 混用在同一个健康度公式里 —— 要么全用 RedStone，要么给 Chainlink 抵押品加一个能覆盖该 feed 最大陈旧度的额外 haircut。

## 四、Solidity 通用篇

### 4.1 `try/catch` 的两个陷阱

案例（[SolvencyFacetProd.sol:84-86](../contracts/facets/SolvencyFacetProd.sol#L84-L86)）—— 意图是"如果这个资产有借贷池，就拒绝用 Chainlink"：

```solidity
try tokenManager.getPoolAddress(symbols[i]) {
    revert BorrowableAssetRedstoneRequired(symbols[i]);   // 有池子 → 拒绝
} catch {}                                                // 没池子 → 放行
```

#### 陷阱 A：`catch {}` 吞掉的是**任何** revert，而失败方向是"放行"

它无法区分：

- ✅ "这个资产没有 pool"（`require` 失败，**预期内**）
- ❌ "TokenManager 被升级坏了 / 地址配错了 / gas 不够"（**预期外**）

**两者都走同一个 `catch {}`，结果都是放行。** 一个本该 fail-closed 的安全检查，实际是 fail-open 的。

#### 陷阱 B：63/64 gas 规则让 catch 可被**主动触发**

这条我之前没意识到。EVM 的外部调用最多只能转发**当前剩余 gas 的 63/64**。所以：

**调用者可以精心控制传入的 gas 总量，使得内层 `getPoolAddress` 恰好 OOG，而外层还剩 1/64 足够继续执行。** OOG 被 `catch {}` 捕获 → 放行。

这意味着 `catch {}` 不只是"可能"意外放行，而是**可以被攻击者按需触发**。在本例中后果是：一个可借资产被静默路由到 Chainlink，**"债务必须用 RedStone"这条不变量被绕过** —— 然后直接接上[第三章](#三-混合预言机篇新鲜度不对等)的攻击。

#### 正确做法

| 做法 | 说明 |
| --- | --- |
| **优先：改成不 revert 的查询** | `getPoolAddressOrZero(sym) == address(0)`，用返回值判断，根本不用 try/catch |
| 必须用 try/catch 时：**匹配具体错误** | `catch Error(string memory reason)` / `catch (bytes memory)`，只吞你预期的那种 |
| **默认方向设为拒绝** | 意料之外的失败应该 revert，不是放行 |
| 检查 gas | 需要严格保证时，`gasleft()` 断言足够 gas 才发起调用 |

> 📌 **通用教训**：`catch {}`（空 catch）在安全检查里几乎总是 bug。写下它的时候要问："我到底想吞掉哪一种失败？其他失败发生时会怎样？"

### 4.2 同一个文件里出现相反的失败哲学 = 设计没想清楚

这是个很好用的**启发式**，值得单独记：

在同一个 `SolvencyFacetProd` 里同时存在：

- **可疑点 1**（staleness）：失败方向 = **锁死整个协议**（过度严格）
- **可疑点 3**（try/catch）：失败方向 = **静默放行**（过度宽松）

**当同一个安全边界上的两个检查，失败哲学相反时，说明作者没有系统性地想过"这个模块该 fail 向哪边"。** 这本身就是"这块还有别的问题"的强信号 —— 值得在这个文件上多花时间。

### 4.3 可见性过大：`public` 的价格入口

`isSolventWithPrices(CachedPrices memory)`、`getHealthRatioWithPrices(...)`、`_getTotalAssetsValueBase(...)` 都是 `public`，**任何人都能拿伪造的价格数组直接调 Diamond**。

在本仓库里这**不是漏洞**：它们都是 `view`、不写状态，真正的 `remainsSolvent` 路径走的是自己构造价格的 `_getAllPricesForLiquidation()`。

**但它建立了一个只靠自觉维持的约束**：任何外部合约都不能信这几个返回值。这个约束没有被任何代码强制，只存在于作者脑子里。

> 📌 **通用教训**：一个"接受价格作为参数"的 `public view` 函数，本质上是**任何人都能操纵返回值的函数**。这类函数命名上应该显式带 `unsafe`/`Trusted` 前缀，或干脆收成 `internal`。集成方（尤其是别的协议）看到 `isSolventWithPrices` 这个名字，很难想到"这个函数的返回值是调用者自己说了算的"。
>
> **注意 `_` 前缀不是可见性**：`_getTotalAssetsValueBase` 带下划线却是 `public` —— 命名约定和实际可见性脱节，会让 review 直接看漏。

### 4.4 死接口

`ISolvencyFacetProd` 声明了 `getMaxDataTimestampDelay()`、`isSignerAuthorized()`、`isTimestampValid()` —— **这三个函数在整个 `contracts/` 里没有任何实现**（RedStone 老版本的遗留）。

本身无害（通过接口调会撞上空 facet 而 revert），但**会误导审计**：读接口的人会以为合约提供了这些自省能力，从而假设"签名者授权是可查询/可验证的"。

> 📌 **通用教训**：接口文件也是攻击面文档的一部分。审计时**别只读接口，要 grep 实现是否存在**。库升级后残留的接口方法是常见噪声源。

## 五、实测验证：方法与本仓库的真实数据

前面几章是"读代码 + 读文档"得出的论断。这一章把其中可验证的部分**在 DeltaPrime 的真实链上部署上跑了一遍**，用的是 Arbitrum / Avalanche 主网数据。目的有二：一是证明这些不是纸上谈兵，二是示范**审计结论应该怎么落地验证**——链上真实状态经常和仓库代码给你的第一印象不一样。

工具：`cast`（Foundry）直接读主网、`anvil --fork-url` 起本地 fork 打真实部署字节码、Chainlink 官方 feed registry、RedStone gateway。全程没有依赖本仓库的 `node_modules`（它没装）。

### 5.1 先问最要命的问题：Chainlink 分支到底启用了没有

**方法**：`setChainlinkFeed` 是 `onlyOwner`，且全仓库的 deploy/test/tools 里**没有任何地方调用它**——所以不能从代码判断，必须去链上逐个 token 查 `getChainlinkFeed`。

```bash
# 对 TokenManager 支持的每个 token 调 getChainlinkFeed，数非零的
cast call $TM "getChainlinkFeed(address)(address)" $token --rpc-url $ARB
```

**实测结果**（TokenManagerTUP，主网）：

| 链 | 支持的 token 数 | 配了 Chainlink feed 的 |
| --- | --- | --- |
| Avalanche | 23 | **0** |
| Arbitrum | 50 | **2** |

Arbitrum 上仅有的 2 个是 GMX 的 GLV 金库代币：

| symbol | token | feed | 是否可借 |
| --- | --- | --- | --- |
| `GLVWETHUSDC` | `0x528A…79F9` | `0xD0E3…8851` | 否（走 Chainlink） |
| `GLVBTCUSDC` | `0xdF03…4b96` | `0xc698…1A85` | 否（走 Chainlink） |

> **教训**：一个 `onlyOwner` 的配置项、代码里搜不到调用，**不代表链上没配**。owner 可以在部署后任意时刻手工设置。判断"某分支是否 live"只能查链上状态，不能靠读部署脚本。

### 5.2 这些是不是真 Chainlink feed？GLV 又是什么？

一开始我怀疑这俩是不是伪装成 Chainlink 接口的自定义合约（Chainlink 不发 LP token 价格吧？）。**实测证否**：

```bash
cast call $FEED "description()(string)" --rpc-url $ARB   # "GLV [WETH-USDC]-USD"
cast call $FEED "aggregator()(address)" --rpc-url $ARB   # 有独立 aggregator，version=6
```

它们是**如假包换的 Chainlink feed**。Chainlink 确实为 GMX 的 GLV（GMX Liquidity Vault，一篮子 GM 头寸）发布 NAV 价格——这是我之前不知道的。而 **RedStone 的 847 个 feed 里没有 `GLVWETHUSDC`**（`curl` gateway 确认）。这就解释了整个 Chainlink 分支存在的**设计动机**：有些资产 RedStone 不报价，只能退到 Chainlink。

> **教训**：混合预言机往往不是架构师"想要"两套，而是**被迫的**——主 oracle 覆盖不全，剩下的窟窿用第二个 oracle 补。而"补窟窿"的那部分资产，恰恰是最容易在新鲜度/校验上被将就的。

### 5.3 GLV 是真抵押品，不是摆设

如果 debtCoverage=0，前面的新鲜度攻击就无从谈起（它对借贷能力没贡献）。**实测**：

```bash
cast call $TM "tieredDebtCoverage(uint8,address)(uint256)" 0 $GLV --rpc-url $ARB
```

| | BASIC 档 | PREMIUM 档 |
| --- | --- | --- |
| GLV[WETH-USDC] | 0.8333e18（**83.3%**） | — |
| GLV[BTC-USDC] | 0.8333e18 | 0.909e18（**90.9%**） |

9 个 Prime Account 里 **4 个持有 GLV**（余额几十美元量级——量小，但攻击面是"活"的）。所以[第三章](#三-混合预言机篇新鲜度不对等)的分子/分母不对称在 Arbitrum 上是**真实可达**的：GLV 抵押品可能陈旧，债务（ETH/USDC 强制 RedStone）永远新鲜。

### 5.4 ⭐ 6 小时常量 vs 24 小时 heartbeat：从"推断"到"实证"

这是全篇最重要的一条，我用两种独立方法验证。

**方法一，查 Chainlink 官方 registry**（比读文档权威，是机器可读的真值）：

```bash
curl -s "https://reference-data-directory.vercel.app/feeds-ethereum-mainnet-arbitrum-1.json"
```

| feed | heartbeat | deviation |
| --- | --- | --- |
| GLV [WETH-USDC]-USD | **86400s = 24h** | 0.25% |
| GLV [BTC-USDC]-USD | **86400s = 24h** | 0.25% |

而合约里的 `CHAINLINK_MAX_DELAY = 6 hours`（注释还写着 `updated to 25 hours`，三个数字互相打架）。**24h heartbeat > 6h 上限**：feed 完全正常也可能超过 6 小时不更新。

**方法二，扫 31 天的 `AnswerUpdated` 事件，实测真实更新间隔**——因为 registry 给的是"规格上界"，我想知道"现实里到底多久不更新一次"：

```bash
# eth_getLogs 扫 aggregator 的 AnswerUpdated，用事件里的 updatedAt 算间隔
topic0 = keccak("AnswerUpdated(int256,uint256,uint256)")
```

| feed | 31 天更新次数 | 间隔中位数 | **最大间隔** | 超 6h 次数 | 超 12h |
| --- | --- | --- | --- | --- | --- |
| GLV[WETH-USDC] | 690 | 0.51h | **17.95h** | **14** | 2 |
| GLV[BTC-USDC] | 444 | 0.83h | **24.01h** | **21** | 6 |

**读数**：

- 中位数半小时更新一次（GLV 跟随 ETH/BTC，0.25% 偏离阈值经常触发），所以**大多数时候没事**——这正是这类 bug 阴险的地方，平时完全看不出来。
- 但 GLV[BTC-USDC] 有一次间隔 **24.01 小时**，正好撞满 heartbeat。那次期间，合约 `block.timestamp - updatedAt > 6h` 成立长达约 **18 小时**。
- 一个月内，6 小时上限被**正常 feed 行为**突破了 14 次（WETH）/ 21 次（BTC）。平均每 1~2 天一次。

**每次突破意味着什么**：任何持有该 GLV 抵押品的账户，其 `getChainlinkPrice` → `getPrice` → `isSolvent()` / `getHealthRatio()` 全部 revert。于是 `remainsSolvent` 修饰器保护的**所有用户操作 revert，清算也 revert**——协议对这些账户被"砖"住十几个小时，期间坏账无人能处理。触发不需要攻击者，市场安静半天就自己发生。

> **教训**：`heartbeat` 是上界，`registry` 给规格，但**只有扫事件才知道现实分布**。"平时半小时更新"会让你以为 6h 常量绰绰有余；只有把 31 天的尾部拉出来，才看到那条 24h 的尖峰。审计陈旧度阈值时，**去扫历史事件求最大间隔**，别只看 heartbeat 配置，更别信"应该不会那么久"。

### 5.5 fork 实测：纯 Chainlink 资产仍然强制要求 RedStone payload

起一个 Arbitrum fork（`anvil --fork-url`），对真实部署的 Prime Account 字节码调用：

```bash
cast call $PA "getPrice(bytes32)(uint256)" $(cast format-bytes32-string GLVWETHUSDC) --rpc-url $fork
# -> revert 0xe7764c9e = CalldataMustHaveValidPayload()
```

即使 `GLVWETHUSDC` 是**纯 Chainlink 定价**、根本不用 RedStone 的价，不带 RedStone payload 也读不出价。印证了 [2.5 节](#25-空-feed-列表--门限检查被完全跳过) 的推断：`getPricesFromRedstoneAndChainlink` 里那条 `getOracleNumericValuesWithDuplicatesFromTxMsg` 无条件执行，`validateTimestamp` 兜底 revert。

**运维含义**：清算人 / 前端对任何资产取价，都必须附带一个新鲜的 RedStone payload——即便该资产的价压根来自 Chainlink。这是 pull-oracle 架构的连带成本。

### 5.6 一个验证过程中撞见的意外：仓库 vendored 的 RedStone 库落后于链上部署

本来想在 fork 上完整复现 6.4 的 staleness revert（构造合法 RedStone payload → 用 anvil 把 GLV feed 替换成"7 小时前"的 mock → 观察 `ChainlinkStaleData`）。payload 我从 gateway 拉真实签名包拼装，并在本地用 `ecrecover` 验证三个签名者都能正确恢复、且都在 0.9.0 库的授权集里。**但打到链上部署的 facet 却一律 `SignerNotAuthorised`**。

关键证据：合约在 revert 里**回显的正是我用的真实签名者地址**（如 `0x51Ce04Be…`）。如果我的 payload 字节布局有偏移 bug，合约恢复出的会是乱码地址；它精确回显了我的签名者，说明**布局是字节级正确的，是链上的授权集合不认这批地址**。

把链上真正执行 `getPrice` 的 facet 字节码拉下来（通过 beacon 的 loupe `facetAddress(0x31d98b3f)` → `0x0D10d390…`，注意这个地址和仓库 `deployments/` 里的 artifact **不同**，说明 facet 已被升级），grep 五个 0.9.0 canonical 签名者——**一个都不在**。它硬编码的是另一组地址。

**结论与教训**：

1. 链上部署授权的 RedStone 签名者集合，与仓库 pin 的 `evm-connector@0.9.0` 里的默认集合**不一致**。最可能是 RedStone 轮换过签名者，而仓库 vendored 的库版本落后于实际部署。大概率是**良性的版本漂移**，不是漏洞——但它意味着：
2. **审计签名者门限这类"信任根"，必须以链上部署字节码为准，不能信仓库里 vendored 的库。** 我正是因为信了 0.9.0 的签名者列表，才在 fork 复现上卡了住。这恰好把 [2.2 节](#22--所有安全钩子都是-virtual-的--攻击面不在你读的文件里) 那条"攻击面不在你读的文件里"演了一遍：真相在链上，不在 `node_modules`。
3. **部署 artifact ≠ 链上字节码**：`deployments/*.json` 记的是历史某次部署，facet 升级后就对不上了。要 `facetAddress(selector)` 从 diamond 现查。

（也因为这个签名者漂移，6.4 的 revert 我最终用 registry + 31 天事件数据实证，而没有在 fork 上跑通那最后一步——需要用链上当前授权的那组签名者的 payload 才行。结论不受影响：`X - Y > 21600` 在实测 X−Y 最大 24.01h 时是算术必然。）

## 六、审计 Checklist

拿到一个预言机集成，按这个顺序过：

### Chainlink 路径

- [ ] staleness 阈值是 **per-feed** 还是全局常量？和每个 feed 的**真实 heartbeat**（去 data.chain.link 查）对得上吗？
- [ ] staleness revert 冒泡上去会挡住**清算**吗？（挡住 = 协议可被砖）
- [ ] 有 `minAnswer`/`maxAnswer` 熔断检查吗？（走 `.aggregator()` 读，别缓存 aggregator 地址）
- [ ] 有没有 `answeredInRound < roundId` 这种**恒为假**的假检查？
- [ ] decimals 归一化后**重新**检查过 `price != 0` 吗？下游有没有 `if (price == 0)` 的宽容分支？

### RedStone 路径

- [ ] **grep 整个继承树**，`getUniqueSignersThreshold` / `getAuthorisedSignerIndex` / `validateTimestamp` / `aggregateValues` 有没有被 override？
- [ ] `mock/` 里的 override 有没有可能进入生产继承链或部署脚本？
- [ ] 跨 facet / 跨合约调用是否都走了 `ProxyConnector`？有没有裸的 `call(abi.encodeWithSelector(...))`？
- [ ] 裸调用的失败是被 revert 还是被 `if (success)` 静默吞掉？
- [ ] 用的是哪个 `getOracleNumericValues*`？如果是 `...AndTimestampFromTxMsg`，**自己验时间戳了吗**？
- [ ] feed 列表能不能被推成**空**？空了之后哪些校验会凭空消失？
- [ ] **有没有在循环里取价？**（gas 二次方 → 清算 DoS）

### 混合路径（最容易漏，优先级最高）

- [ ] **把所有价格来源的新鲜度窗口列成一张表**，量级差多少？
- [ ] 路由规则（"债务必须用 X"之类）是否让偏差**单向化**？
- [ ] 偏差落在健康度公式的分子还是分母？攻击者能选时机吗？

### 通用

- [ ] 有 `catch {}` 空 catch 吗？失败方向是放行还是拒绝？能被 63/64 gas 规则主动触发吗？
- [ ] 同一模块里不同检查的**失败哲学一致**吗？（一个锁死、一个放行 = 危险信号）
- [ ] 接受价格作为参数的 `public view` 函数，命名有没有暴露"返回值不可信"？
- [ ] 接口里声明的方法**都有实现**吗？

## 附录：怎么读一个没装的依赖的源码

本仓库的 `node_modules` 没装，但审计必须读 RedStone 的**真实**源码 —— **不能靠回忆，版本之间差异很大**（比如 0.9.0 的默认门限、`validateTimestamp` 的具体窗口，都是版本相关的）。

不跑完整 `yarn install`（这个项目装依赖很重，还要访问 GitHub 直装包）的办法：

```bash
# 从 package.json 拿到确切版本："@redstone-finance/evm-connector": "0.9.0"
cd /tmp/scratch
npm pack @redstone-finance/evm-connector@0.9.0
tar xzf redstone-finance-evm-connector-0.9.0.tgz
find package/contracts -name "*.sol"
```

**关键是版本号必须从 `package.json` / `yarn.lock` 里抄**，不是装 latest。审计的是部署的那个版本。

这次要读的核心文件（对任何 RedStone 集成都一样）：

| 文件 | 看什么 |
| --- | --- |
| `core/RedstoneConsumerBase.sol` | 所有 `virtual` 安全钩子；`_securelyExtractOracleValuesAndTimestampFromTxMsg` 主流程；`_getAggregatedValues` 的门限检查 |
| `core/RedstoneConsumerNumericBase.sol` | `getOracleNumericValues*` 各变体的区别（**哪个不验时间戳**） |
