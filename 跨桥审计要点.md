# 跨桥审计要点（通用 · 可带走）

这份文档**不绑定任何项目**，是《赏金审计开局 checklist》的 **跨桥换血包**：checklist 的元流程（读范围 → 信任边界 → 找兜底 → 假设排序 → 四问闸门 → 报告）不变，只把**阶段 2 的兜底目录**、**阶段 0 的排除项**、**阶段 3 的类型学**换成跨桥味的。

> 用法：打桥项目时，checklist 阶段 2 那句"桥请用跨桥目录替换本节"→ 就是指向本文《§2 兜底目录》。元动作"**先找兜底再决定挖不挖**"一字不变。
>
> 本文的具体样本来自一次 HyperBridge（Polkadot 系 · BEEFY + ISMP）审计，但目录本身对 LayerZero / Wormhole / Axelar / 乐观桥 等同样适用 —— 换的是"信任根长什么样"，不换"该问哪几个问题"。

---

## 为什么单列一份

桥和借贷/DEX 的**兜底形态不同**，但会撞死人的逻辑是同一套：

- 借贷靠 **预言机计价 + 余额差 + 偿付性**。
- DEX 靠 **K 不变量 + reserve/balance 对账 + 用户滑点**。
- **桥靠 `门限签名（信任根）+ 消息去重（防重放）+ 守恒（托管进出配平）`**，把"伪造/重放/冒领消息"三类压死。攻击面被压缩到只剩**信任根本身**、**没有兜底罩着的面**、以及**两条链之间的时序缝**。

> **核心命题**（与 checklist 一致）：当桥用"门限签名验证的状态根 + 全字段承诺去重 + 对等认证守恒"罩住消息通道时，它把 calldata 层的伪造全降级成"需要 validator 合谋"。**在这三面上找 calldata 花招 = 白挖。** 真正的活口在别处（下 §2/§3）。

---

## §0 阶段 0 换血：跨桥高频排除项（雷区）

任何一条命中，对应整类攻击面**直接作废**。读范围时逐条比对。

| 排除项 | 作废掉什么 | 备注 |
| --- | --- | --- |
| **temporary freeze / liveness / DoS** | 所有"卡住消息、冻结托管、relayer 不作为"类 | ⚠️ 桥的失效模式天生一大半是 revert/卡住。**这条在不在，决定半个攻击面生死** |
| **requires malicious/colluding validators** | 所有"如果 2/3 validator 作恶"类 | 信任根级伪造几乎都撞这里 → 除非是**不需恶意 validator 的密码学 bug** |
| **requires privileged relayer / trusted relayer** | 依赖 relayer 作恶的 | 看 relayer 是否 permissionless |
| **front-run-only** | 仅靠抢跑触发的 | 通用 |
| **third-party / imported contracts** | vendored 的证明库、OZ 等 | ⚠️ 陷阱：桥的信任根常 vendored 一堆库，**先确认库在不在 in-scope repo 里**，模糊时问 |
| requires external chain reorg / finality failure | "如果源链回滚" | 常 out，但**乐观桥的挑战期**相关可能 in |
| already known | 桥的公开报告极多 | 先搜 |

### 阶段 0 产出（跨桥版一句话）

> **"我要找的是：`不可信的 relayer / user / solver`，能通过 `消息中继 / intent 订单 / dispatch` 这类入口，`窃取或损失资金 / 伪造未授权跨链消息 / 篡改交易`，且不依赖 validator 合谋、不依赖抢跑。"**

## §1 阶段 1 换血：跨桥信任边界（桥的本质）

桥的特殊性，一句话记死：

> **调用者可信 ≠ 数据可信。** 在 `onlyHost` / `onlyEndpoint` 回调里，caller 是本地 host（可信），**但 payload 是远端链上任意人撰写的**。caller 校验只保证"确实过了证明通道"，不保证"内容善意"。

三类边界分清（别看修饰器名字，看它验的是证明还是身份）：

| 边界 | 长什么样 | 结论 / 挖点 |
| --- | --- | --- |
| **治理侧** | `restrict(admin)` / `restrict(hostManager)` / `onlyRole` | 🚫 可信，别挖（除非能证明治理消息可被非治理方伪造经证明通道送达） |
| **信任根 / Handler** | `handleConsensus` / `handlePostRequests` 等**无角色修饰器，任何人可调** | 🎯 安全性 = **证明验证**，不是 caller 身份。挖点全在"证明能否被伪造/绕过" |
| **应用回调** | `onAccept` / `onGetResponse`（`onlyHost`） | 🎯 **caller 可信 / 数据不可信**：source chain、sender、body 由远端撰写 → 对等认证、commitment 绑定、重放 |
| **用户/solver 直连** | `dispatch` / `placeOrder` / `fillOrder` / `cancelOrder` | 🎯 托管锁定、费用记账、状态机一致性、跨链竞态 |

> ⚠️ **同名修饰器双语义陷阱**（通用）：`onlyOwner` 可能是协议 owner（可信）或账户 owner（普通用户=攻击者）。**看它读谁的存储，不看名字。**

**产出**：入口表，只在"信任根 / 应用回调 / 用户直连"三类不可信行继续。

## §2 阶段 2 换血：⭐ 跨桥兜底目录（本文最值钱的一节）

### 🔑 桥兜底最值钱的一问

> **哪些字段进了签名 / proof 的 preimage，哪些没进？**

桥的消息承诺通常 = `keccak256(abi.encode(Message))`。**逐字段核对 preimage 是否覆盖全部安全相关字段**：`source · dest · nonce · from · to · timeout · body`。**任何漏进 preimage 的字段 = 可被篡改而承诺不变 = 头号高危。** 这是桥审计第一刀，纯读代码，💰 最便宜。

### 三层兜底：一条伪造消息要穿几道墙

```
攻击者想伪造 "放我托管 / 执行我的调用" 的消息：

墙1  门限签名（信任根）      验 2/3+1 supermajority + 签名恢复 + 防重复签名者
      └► 穿墙成本 = 伪造 validator 签名 = 合谋（死结，别挖）
墙2  proof + 去重 + 目的地绑定  dest==本链 + 状态证明(MMR/Trie) + receipts 去重
      └► 穿墙 = 让 committed 状态根认可一条不存在的消息
墙3  应用层对等认证 + 守恒    from==源链注册对等实例 + 托管下溢守恒 + 先交付后领取
      └► 穿墙 = 绕过 _authenticate 或撬开守恒
```

### 兜底目录

| 兜底 | 长什么样 | 它罩住了什么 |
| --- | --- | --- |
| **门限签名 / 信任根** | 2/3+1 supermajority、`ecrecover` 防延展/零地址、多证明拒重复签名者 | 伪造共识 |
| **全字段承诺 + 去重** | `commitment=keccak(全字段)`、`requestReceipts[hash]!=0 → 拒` | 篡改字段、同链/跨链重放 |
| **目的地绑定** | `request.dest == host.host()` | 跨链错投 / 把消息重放到别的链 |
| **对等认证** | `from == _instance(source)`，instances 只能由治理消息写 | 冒充源链网关冒领托管 |
| **托管守恒** | `_orders[c][token] -= amount`（0.8 下溢 revert）、`balanceOf` 差值记账 | 超额提取、fee-on-transfer 偏移 |
| **先交付后领取** | fill 先转 output 给受益人，再 dispatch redeem | solver 空手套托管 |
| **重入锁** | `nonReentrant` on place/fill/cancel | 重入 |

### 判定规则

1. **先找兜底，再决定挖不挖。** 三面（伪造/重放/冒领）都罩着 → 别在这找 calldata 花招。
2. ⚠️ **如果穿墙唯一路径是"validator 合谋"且它 out-of-scope → 这面是死结，早撤。**
3. 去打**兜底没覆盖的面**（下 §3）。

## §3 阶段 3 换血：跨桥逻辑类型学

checklist 通用类型学照用（Decode/Execute 分歧、Fail-open on zero、不对称检查、单向校验、空 catch、可信角色边界）。补这几条跨桥专属，**按证伪成本从低到高**：

| 类型 | 模式 | 去哪找 | 证伪成本 |
| --- | --- | --- | --- |
| **字段漏进 preimage** | 安全相关字段未进 commitment/签名哈希 | 消息 encode/hash 函数 | 💰 读代码（第一刀） |
| **source-machine 认证不全** | 只认"是条消息"，不认"来自哪台注册机器" | `onAccept` 的 `_authenticate` | 💰 读代码 |
| **跨链时序竞态** | cancel(源) 与 fill(目的) 的高度/deadline 窗口 | 两条链高度不同源处 | 💰 纯推理（算高度窗口） |
| **非成员证明 gating 资金 + 空值键** | timeout/退款要"证明目标链无 receipt"，但判定用 `value.length==0` 而非 `keyPresent` | timeout / refund 路径 | 💰💰 读代码 + 追存储值类型 |
| **decode/execute 分歧** | 校验解码后视图，却转发原始字节给外部 | router 包装、`call(encodePacked(...))` | 💰💰 读调用链 |
| **证明验证器解码发散** | 手写 SCALE/RLP/trie 解码器接受非规范编码，得出规范解码器不会产出的 key/value | trie 节点头、compact 长度解析 | 💰💰💰💰 读+查规范 |
| **门限计数 vs distinct** | 阈值数 `signatures.length` 而非 distinct signer | 共识验证器的 supermajority 检查 | 💰💰 读代码 |

### 信任根深挖的两条真实教训（来自 HyperBridge）

**教训 1 —— 哈希锚定使"伪造"退化成"碰撞"。** MMR / MultiProof / Trie 验证的 root 都由上游共识固定，每个节点要 `keccak(preimage)==committed_hash` 才被接受，子节点随父锚定。**所以信任根的 bug 不是"纯伪造"，只可能是"逻辑让非规范输入算出相同 root"** —— 只查这类口：
- 门限用 `votes.length` 还是 distinct？（重复签名凑门限）→ 看 leaves 是否按 index 严格排序拒重复
- 未平衡树的"promote unchanged"、空证明、重复叶子 → 看边界 invariant
- 解码器发散（上表最后一行）→ 成本最高、命中率最低，最后再碰

**教训 2 —— 非成员证明 + 空值键，是桥特有的"库对、调用方错"陷阱。** ⭐ **强复用**
> timeout / 退款路径常要"证明目标链的 receipt 存储里**没有**这条 commitment"（非成员），才放行退款。
> 若调用方用 `value.length == 0` 判定"不存在"，而底层链（尤其 substrate/FRAME）**允许存空值键**（`()` 表集合成员），就有缝：
> **消息实际已投递（键存在、值为空）→ `value.length==0` 被误判为"未投递"→ 骗过 timeout 拿退款 → 已执行 + 退款 = 双花。**
>
> 证伪要追到底层存储值类型：若 receipt 存的是**非空**（relayer 地址 / 结构体，SCALE 编码 ≥1 字节）→ present ⟺ 非空 → 攻击不成立。若能找到**任一非成员证明 gating 了资金、且对应存储项存 `()`/空值 → 真 critical。**
> 审计动作：grep 所有非成员判定（`value.length==0` / `!contains` / `keyPresent==false`），逐个追它 gating 什么、对应存储值是否可为空。

## §4 阶段 4 闸门（跨桥版四问，不变）

1. **攻击者是谁？** 必须不可信（relayer/user/solver）。**不能依赖 validator 合谋**（若排除）。
2. **失效模式是什么？** 必须是错误数值/状态。是 revert/卡住 → 看是否撞 temporary-freeze 排除。
3. **钱从谁兜里出？** 必须是 LP / 协议托管 / 其他用户。是 solver 自承风险、无攻击者获利 → 可能撞 griefing，定级弱。
4. **属于哪一类 in-scope？** 说不出来就停。

## §5 报告 + 工具箱（复用 checklist，跨桥补丁）

| 手法 | 说明 |
| --- | --- |
| **信任根从链上字节码确认** | vendored 库 ≠ 部署时链接的库；签名者/门限/validator set **必须查链上**，别信仓库常量 |
| **签名 payload 从真实交易扒** | 需要 BEEFY/validator 签名但没私钥 → 从真实中继交易 calldata 尾部扒，fork 到那个区块（时间戳/签名者才匹配） |
| **叙事绕损失路径** | 别写成"证明验证函数会 revert"，写成"伪造消息 → 冒领托管 → LP 亏损" |
| **诚实定级** | 主动说出减分项（自愈、条件性、无攻击者获利、需特定 token） |

| | 借贷 | **桥** | DEX |
