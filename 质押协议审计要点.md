# 质押协议审计要点（通用 · 可带走）

这份文档**不绑定任何项目**，是《赏金审计开局 checklist》的 **流动性质押（LST）换血包**：checklist 的元流程（读范围 → 信任边界 → 找兜底 → 假设排序 → 四问闸门 → 报告）不变，只把**阶段 2 的兜底目录**、**阶段 0 的排除项**、**阶段 3 的类型学**换成质押味的。

> 用法：打质押/再质押项目时，checklist 阶段 2 那句"请用对应目录替换本节"→ 就是指向本文《§2 兜底目录》。元动作"**先找兜底再决定挖不挖**"一字不变。
>
> 本文的具体样本来自一次 LiNEAR（NEAR 流动性质押 · epoch actions + staking-pool 回调）审计，但目录本身对 Lido / RocketPool / Stader / EigenLayer 等同样适用 —— 换的是"份额价格的分子从哪来、验证人余额谁上报"，不换"该问哪几个问题"。

---

## 为什么单列一份

质押和借贷/DEX/桥的**兜底形态不同**，但会撞死人的逻辑是同一套：

- 借贷靠 **预言机计价 + 余额差 + 偿付性**。
- DEX 靠 **K 不变量 + reserve/balance 对账 + 用户滑点**。
- 桥靠 **门限签名 + 消息去重 + 守恒**。
- **质押靠 `份额价格单调不减 + 记账↔真实余额对账 + 释放窗口/取款闸`**，把"取整套利、记账漂移、提前取款"三类压死。攻击面被压缩到只剩**份额价格的分子来源（谁上报验证人余额）**、**没有对账罩着的路径**、以及**epoch/withdrawal 状态机的时序缝**。

> **核心命题**（与 checklist 一致）：当 LST 用"铸/销份额时取整方向永远利协议 + 份额价格永不下降"罩住用户资金面时，它把 calldata 层的取整花招全降级成"最多漏 dust，且价格只升不降"。**在这面上找取整套利 = 白挖。** 真正的活口在别处：**喂假余额抬高价格分子、绕过对账、状态机卡死**（下 §2/§3）。

## §0 阶段 0 换血：质押高频排除项（雷区）

任何一条命中，对应整类攻击面**直接作废**。读范围时逐条比对。

| 排除项 | 作废掉什么 | 备注 |
| --- | --- | --- |
| **temporary freeze / liveness / DoS** | 所有"卡住 epoch、冻结取款队列、validator 不响应"类 | ⚠️ 质押的失效模式天生一大半是 revert/卡住（validator 是 pull 式外部调用）。**这条在不在，决定半个攻击面生死** |
| **requires external protocol failure** | "如果 staking pool / beacon chain 挂了"、"如果 validator 上报假余额" | ⚠️ **质押的价格分子来自外部验证人余额** —— 大半高危活路撞这里 |
| **slashing / consensus penalty** | 所有"余额下降致下溢/坏账"类 | 很多链 slashing 未启用（如 NEAR），相关 bug 变 latent；EVM 上 Lido 有 oracle 上报负 rebase |
| **requires privileged/admin role** | validator 白名单、oracle 委员会、manager 设 base_stake/weight | validator 通常 owner/manager 白名单化 → rogue validator 不在威胁模型 |
| griefing / no attacker profit | "让协议难受但攻击者不获利"（如把 stake 导向差 validator） | 通用 |
| **front-run-only** | 仅靠抢跑 permissionless epoch 动作触发的 | ⚠️ 但"frontrunning"有时明确 in-scope，看清单 |
| already known | LST 公开报告多（Lido/Rocket 审计全公开） | 先搜 |

### 阶段 0 产出（质押版一句话）

> **"我要找的是：`不可信的 user / permissionless keeper`，能通过 `deposit/stake/unstake/withdraw / epoch 动作` 这类入口，`让份额价格被高估或下降、窃取其他持有人价值、或提前/超额取走资金`，且不依赖 validator 作恶、不依赖 slashing、不只是让协议难受。"**

## §1 阶段 1 换血：质押信任边界

LST 的特殊性，一句话记死：

> **调用者可信 ≠ 数据可信。** 在 `validator_*_callback` / `oracle report` 回调里，caller 是合约自身或 oracle（可信），**但 payload 是外部验证人/信标链上报的余额**。caller 校验只保证"确实是回调"，不保证"余额真实"。

四类边界分清（别看修饰器名字，看它验的是身份还是外部数据）：

| 边界 | 长什么样 | 结论 / 挖点 |
| --- | --- | --- |
| **治理/管理侧** | `assert_owner` / `assert_manager` / `onlyRole`：设 validator、weight、base_stake、fee、oracle 成员 | 🚫 可信，别挖（validator 白名单化后，"rogue validator"通常也随之出范围） |
| **permissionless keeper** | `epoch_stake/unstake/update_rewards/withdraw`、`rebase`、`report`（**无角色修饰器，任何人可调**） | 🎯 挖点：候选选择能否被操纵、能否 double-trigger、能否卡死 settle、抢跑窗口 |
| **外部余额回调** | `validator_get_balance_callback` / oracle `submitReport`（`#[private]` / `onlyOracle`） | 🎯 **caller 可信 / 数据不可信**：上报的 staked/unstaked/reward 余额是否**有对账兜底**、有无上限、能否单向 |
| **用户直连** | `deposit / stake / unstake / withdraw / ft_transfer(_call)` | 🎯 份额↔资产换算取整方向、取款解锁 epoch、注册/注销边界 |

> ⚠️ **同名修饰器双语义陷阱**（通用）：`onlyOwner` 可能是协议 owner（可信）或账户 owner（普通用户=攻击者）。**看它读谁的存储，不看名字。**（NEAR 上账户身份即 `predecessor_account_id`，无此陷阱；EVM/vault 上要警惕。）

**产出**：入口表，只在"permissionless keeper / 外部余额回调 / 用户直连"三类不可信行继续。

## §2 阶段 2 换血：⭐ 质押兜底目录（本文最值钱的一节）

### 🔑 质押兜底最值钱的一问

> **份额价格 `P = 总质押资产 / 总份额` 的分子从哪来？哪条路径能抬高它、哪条能让它下降？谁给分子喂数、有没有对账罩着？**

LST 的一切安全性都绕着这个价格转。**逐条追分子的每个增减来源**：deposit（用户真金）、reward（外部验证人上报）、slash（外部下降）、fee mint（稀释份额）。**任何"外部上报直接进分子、且不过对账"的路径 = 头号高危。** 这是质押审计第一刀，纯读代码，💰 最便宜。

### 三层兜底：一次取整/抽取要穿几道墙

```
攻击者想"反复 stake/unstake 从其他持有人抽价值 / 提前取走资金"：

墙1  份额价格单调不减保证       铸份额时 total 向上取整入账、销份额时 total 向下取整出账
      └► 穿墙 = 让 P 下降（掏空 price-guarantee fund，通常 dust 级、成本>收益）
墙2  记账↔真实余额对账           post-action 拉 validator 真实余额，|diff| ≤ 容差 才接受
      └► 穿墙 = 找一条不走对账、直接信任外部上报的分子入口（如 reward 路径）
墙3  释放窗口 + 取款三重闸        pending_release 锁 + 解锁 epoch + 合约储备下限
      └► 穿墙 = 绕过解锁 epoch 或掏到储备线以下
```

### 兜底目录

| 兜底 | 长什么样 | 它罩住了什么 |
| --- | --- | --- |
| **份额价格单调不减保证** | stake：charge 向下取整、`total += 向上取整`；unstake：receive 向上取整、`total -= 向下取整`，差额由 guarantee fund 垫 | 所有 deposit/stake/unstake 取整/顺序花招 —— 价格数学上**永不下降** |
| **记账↔真实余额对账（容差）** | post-action 拉 `get_account` 真实余额，三项 diff 全 ≤ `MAX_DIFF` 才接受，否则丢弃保留旧值 | stake/unstake 后的记账漂移（记账 vs 真实） |
| **释放窗口锁** | `pending_release`：解锁期内禁止再 unstake/withdraw 该 validator | 过早二次 unstake、锁定期内提款 |
| **取款三重闸** | unstaked 足够 **且** 解锁 epoch 到 **且** 合约余额 − 储备下限 ≥ amount | 提前/超额/掏空式取款 |
| **per-validator 执行互斥锁** | `pre_execution` 断言 `!executing` 再置 true，回调 `post_execution` 清 | 同 validator 并发/重入 epoch 动作 |
| **份额退款（NEP-141 / ERC4626）** | `resolve_transfer` 按 `min(balance, unused)` 退还 | transfer_call/deposit 回调失败退款 |
| **溢出保护** | `checked_*` + 全局 `overflow-checks` | 记账溢出（但下溢会 **panic** → 见 §3 单向校验） |

### 判定规则

1. **先找兜底，再决定挖不挖。** 用户资金面（deposit/stake/unstake/withdraw）被墙1+墙3 罩着 → 别在这找取整花招。
2. ⚠️ **如果唯一活路是"喂假 validator 余额"且 validator 白名单化 / external failure out-of-scope → 这面是死结，早撤。**（LiNEAR 实测正是此结局。）
3. 去打**兜底没覆盖的面**：不走对账的分子入口（reward 路径）、状态机时序缝（下 §3）。

### 核心算术检查（不查链就杀掉一整类）

> 设价格 `P = total_staked / total_share`。若每步取整方向都利协议（铸时分子多加、销时分子少减）⇒ **P 单调不减** ⇒ 攻击者反复 stake/unstake 只把 dust 送进 guarantee fund，**永远碰不到其他持有人**。
> **唯一活路**：① 让分子被**高估**（喂假 reward 余额）；② 找到让 **P 下降**的路径（绕过 guarantee fund）。**不用查链就能杀掉整片取整套利假设。**

## §3 阶段 3 换血：质押逻辑类型学

checklist 通用类型学照用（Decode/Execute 分歧、Fail-open on zero、不对称检查、单向校验、空 catch、可信角色边界）。补这几条质押专属，**按证伪成本从低到高**：

| 类型 | 模式 | 去哪找 | 证伪成本 |
| --- | --- | --- | --- |
| **取整方向反了** | 铸份额向上取整/销份额向下取整 —— 与"永远利协议"相反，价格可被压低 | `num_shares_from_amount` / `amount_from_shares` 的 rounded_up/down 配对 | 💰 读代码（第一刀） |
| **不对称检查：分子入口漏对账** | reward/rebase 路径直接信任外部上报余额进分子，**不走** stake/unstake 那套对账 | `update_rewards` / `on_report` 回调 | 💰 读代码 |
| **单向校验：余额只增假设** | `rewards = new − old` 无 `new < old` 处理 → 下溢 panic / 或 rebase 负值未处理 | 奖励差值计算处 | 💰 纯推理（余额下降=slash 是否可达） |
| **首存/份额膨胀（inflation）** | 空池首个存款者操纵 `total_share=0/1` 抬价，后来者被四舍五入吞 | 初始化、`total_share==0` 分支、是否有 bootstrap 种子份额 | 💰💰 读代码 + 算 |
| **取款解锁 epoch off-by-one** | cleanup 与 unstake 同 epoch 时 `+1` 边界，算错解锁高度 | `unstaked_available_epoch_height` 计算 + `last_settlement_epoch` 交互 | 💰💰 读代码追跨 epoch |
| **候选选择可被操纵** | 一笔 stake/unstake 改 `total_staked` → 改 target → 抢跑 keeper 把资金导向特定 validator | `get_candidate_to_*` + `target_stake_amount` | 💰 纯推理（能否获利，多半是 griefing） |
| **回调 panic 卡死状态锁** | 回调 `unwrap`/下溢 panic → `executing`/lock 永不清 → 该 validator 永久冻结 | 所有"SHOULD NOT PANIC"的回调里的 `unwrap`/减法 | 💰💰 读代码（触发条件多需外部/slash） |
| **base_stake 比例缩放漂移** | 每次余额更新按比例取整缩放 base_stake，累积漂移扭曲分布 | `sync_base_stake_amount` 类 | 💰💰💰💰 fork 量化（多为 griefing，缓做） |

### 质押深挖的两条真实教训（来自 LiNEAR）

**教训 1 —— 份额价格单调不减保证会把整片取整假设一次杀光。** ⭐ **强复用**
> 一旦确认"铸份额时 `total_staked += 向上取整`、销份额时 `total_staked -= 向下取整`"，价格 P 数学上永不下降。**所有"反复 stake/unstake 从别人抽价值"的假设，纯算术即可证伪，零链上开销。** 别再对着取整写 PoC。
> 转而只问一句：**有没有一条路径能让 P 下降或分子被高估？** 有 → 深挖；没有 → 用户资金面整体结项。

**教训 2 —— 分子的外部来源 + validator 白名单，常构成"死结型"活路。** ⭐ **强复用**
> LST 里唯一能抬高价格分子的，通常是 reward/rebase 路径，而它**信任外部验证人/oracle 上报的余额**（往往不走 stake/unstake 那套对账兜底）。这看着像头号高危 —— 但：
> - 若 validator **白名单化**（owner/manager 加 + whitelist 校验）→ 喂假余额需 rogue validator = external failure = **out-of-scope**。
> - 若余额**下降**触发下溢 panic → 需 **slashing**（很多链未启用）+ 失效模式是 **freeze/DoS**（常排除）。
>
> **证伪动作**：追 validator 准入链（`add_validator → whitelist 校验`）确认威胁模型；追分子入口是否有对账/上限。**若唯一活路落在白名单/external/slash/DoS 任一被排除项 → 早撤，别写 PoC。** 这正是 checklist"兜底/活路恰好落在被排除那一类=死结"的质押版实例。

## §4 阶段 4 闸门（质押版四问，不变）

1. **攻击者是谁？** 必须不可信（user / permissionless keeper）。**不能依赖 validator 作恶或 manager 特权**（若排除）。
2. **失效模式是什么？** 必须是错误数值/状态（价格被操纵、份额多铸、资金错发）。是 revert/卡住 → 看是否撞 temporary-freeze / DoS 排除。
3. **钱从谁兜里出？** 必须是其他 LiNEAR/stETH 持有人 / 协议储备。是攻击者自有权益、或 dust 级 guarantee fund → 停。
4. **属于哪一类 in-scope？** 说不出来就停。

## §5 报告 + 工具箱（复用 checklist，质押补丁）

| 手法 | 说明 |
| --- | --- |
| **份额价格从链上真值确认** | `total_staked_near_amount` / `total_share_amount` 及 validator 数量、base_stake 配置，只能查链上状态，别信仓库常量 |
| **扫翻转点求解锁窗口/阈值** | `NUM_EPOCHS_TO_UNLOCK`、储备下限等常量，用扫事件/翻转点法确认真实值，别只读字节码常量 |
| **fork 时喂受控 validator mock** | 需验证人回调但不便触真实池 → `anvil_setCode`/mock staking-pool 换掉外部依赖（注意：喂假余额若属 external failure，PoC 也不 in-scope，先算 scope） |
| **叙事绕损失路径** | 别写成"reward 回调会下溢 revert"，写成"validator 余额下降 → 回调 panic → executing 卡死 → 该 validator 永久冻结 → 用户无法从其取款"（若 freeze 不排除时） |
| **诚实定级** | 主动说出减分项（自愈、需 slashing、条件性、dust 级、无攻击者获利、需白名单验证人作恶） |

## 附：四类项目兜底目录速查（换血对照）

| | 借贷 | 桥 | DEX | **质押（LST）** |
| --- | --- | --- | --- | --- |
| 兜底 1 | 预言机计价 | 门限签名（信任根） | K 不变量 `x·y≥k` | **份额价格单调不减（取整利协议）** |
| 兜底 2 | 余额差后置校验 | 消息去重（全哈希承诺） | reserve/balanceOf 对账 | **记账↔真实余额对账（容差）** |
| 兜底 3 | 偿付性 `HR≥1` | 守恒 + 对等认证 | 用户滑点 + 舍入方向 | **释放窗口锁 + 取款三重闸 + 执行互斥锁** |
| 头号活路 | 预言机高估清算价值 | 漏进 preimage 的字段 / 非成员证明+空值键 / 时序缝 | 舍入反向 / 特殊 token 绕 K | **喂假 reward 余额抬高价格分子 / 取整方向反了让 P 下降 / 首存份额膨胀 / 解锁 epoch off-by-one** |
| 常见死结 | 预言机 in out-of-scope | validator 合谋 / freeze in out-of-scope | sandwich/MEV/IL in out-of-scope | **rogue validator（白名单化）/ slashing 未启用 / freeze·DoS in out-of-scope** |
