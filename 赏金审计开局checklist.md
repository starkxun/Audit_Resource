# 赏金审计开局 Checklist（通用 · 可带走）

这份文档**不绑定任何项目**，是拿到一个新赏金项目后按顺序执行的开局流程。

## 为什么有这份文档

两轮 DeltaPrime 审计的复盘。两轮都产出了技术上扎实的东西：

- 第一轮：根因定位到常量、链上实测 31 天数据、在**真实部署字节码**上端到端复现清算阻断、两层 PoC、诚实定级
- 第二轮：四条逻辑类假设，逐条推理证伪

**两轮都没能提交成功。** 原因不是技术不够，是**开局顺序错了** —— 把 100% 精力花在"这个 bug 真不真"，0% 花在"这个 bug 能不能提"，直到提交才发现后者是零。

> **核心命题**：一个 bug 的**技术质量**和**可提交性**是两个正交维度。开局阶段要做的，是在写第一行 PoC 之前，把后者算清楚。

---

## 阶段 0：读范围（在读任何代码之前）

**这是最贵的一课。** 第一轮我们做完了完整 PoC 才发现方向必死 —— 而读五分钟范围清单就能杀掉它。

### 要抄下来的三样东西

去项目的 program 页面（HackenProof / Immunefi / Cantina 等），把这三样**抄在纸上贴屏幕边**：

| 抄什么 | 为什么 |
| --- | --- |
| **in-scope impact 清单** | 你的发现必须能指名其中一条，否则一文不值 |
| **out-of-scope / exclusions 清单** | ⚠️ **比 in-scope 更重要**，这里埋着地雷 |
| **assets in scope**（合约地址/链） | 挖了不在范围里的合约 = 白挖 |

### 高频排除项（看到就要警觉）

这几条是最常见的"地雷"，任何一条命中，对应的**整类攻击面直接作废**：

| 排除项 | 作废掉什么 |
| --- | --- |
| **temporary freeze of funds** | ⚠️ 所有 DoS / liveness / 可用性问题。**预言机相关的发现大半死在这里**（pull oracle 的失效模式天生是 revert） |
| griefing / no attacker profit | 所有"攻击者不获利，只是让别人难受"的发现 |
| requires privileged/admin role | 所有需要 owner / 白名单角色作恶的发现 |
| centralization risk / admin key | 所有"如果 owner 作恶"类 |
| gas optimization / best practice | 代码质量类 |
| already known / previously reported | 先搜 public reports |
| requires external protocol failure | "如果 Chainlink 挂了" |
| theoretical / no PoC | 无法复现的推理 |

### 阶段 0 的产出

一句话，写下来：

> **"我要找的是：`<某类角色>` 能通过 `<in-scope 的某一类>`，让 `<某个受害方>` 损失资金。"**

这句话是后面所有工作的过滤器。任何不匹配它的线索，**直接跳过，无论多有趣**。

> 📌 **每读一段代码就问**："如果这里坏了，坏成什么样？属于哪一类？"
> 答案是"**会 revert / 会卡住**"的 → 在有 temporary-freeze 排除的项目里，**立刻放弃**。

## 阶段 1：画信任边界图

第二轮的教训。剩下的线索全部需要"恶意清算人"，而清算人是 owner 精选的白名单 = 可信角色 = 不在威胁模型内 → 全军覆没。

### 做法

grep 出所有**外部入口**及其修饰器，分成三栏：

```bash
grep -rn "function .*external\|function .*public" --include=*.sol contracts/ \
  | grep -v "/mock/\|/test/"
```

| 谁能调 | 结论 |
| --- | --- |
| 协议 owner / governance / 白名单角色 | 🚫 **可信，别挖** |
| **用户自己的账户 owner** | ⚠️ **不可信，重点挖** |
| 任何人（无修饰器） | ⚠️ **重点挖** |
| 外部协议回调（keeper / router callback） | ⚠️ **重点挖**（调用者校验往往有洞） |

### ⚠️ 陷阱：同名修饰器可能有两种语义

DeltaPrime 里 `onlyOwner` 有**两个完全不同的含义**：

- `TokenManager.onlyOwner` → **协议 owner**，可信
- `PrimeAccount.onlyOwner` → **账户 owner，即普通用户**，是攻击者

**别看名字，看它读的是谁的存储。** 这个区分决定了 `paraSwapV6` 是"可信角色"还是"主攻方向"。

### 阶段 1 的产出

一张表：入口 → 谁能调 → 是否可信。**只在"不可信"那些行上继续。**

## 阶段 2：⭐ 找兜底机制（本次最值钱的一条）

**在挖任何面之前，先问：这个面上有没有一个健壮的后置校验罩着它？**

第二轮四条假设全部撞死在同一堵墙上：

```
无论 calldata 怎么构造、路由走哪个池、executor 是谁
        ↓
后置校验：用预言机价格核对真实到账余额  →  每笔最多漏 X%
        ↓
偿付检查：remainsSolvent 在 HR = 1 截停  →  抵押品 ≥ 债务
        ↓
攻击者累计抽走 ≤ 自有权益              →  受害方分文未损
```

### 核心洞察

> **当一个协议在关键路径上用"预言机计价 + 余额差"做后置校验时，它把所有 calldata 层的逻辑漏洞都降级成了"最多漏 X%"，而 X% 又被偿付检查兜住。攻击面被压缩到只剩兜底机制本身。**

### 常见的兜底机制（识别它们）

| 兜底 | 长什么样 | 它罩住了什么 |
| --- | --- | --- |
| **余额差后置校验** | `bought = balanceOf(after) - balanceOf(before)`，再断言 | 所有"骗过参数校验"的花招 —— 因为它测的是**真实结果**，不是声明 |
| **预言机计价的滑点上限** | `if (soldValue > boughtValue) { slippage ≤ MAX }` | 把任意路由/自建池的价值抽取压到 ≤ MAX% |
| **偿付性不变量** | `remainsSolvent` / `healthFactor ≥ 1` 修饰器 | 把累计抽取压到"≤ 攻击者自有权益" |
| **精确授权 + 用后撤销** | `safeApprove(x, exact)` … `safeApprove(x, 0)` | 外部合约最多拉走 `exact` |
| **重入锁** | `nonReentrant` | 重入类 |

### 判定规则

1. **先找兜底，再决定挖不挖。** 一个面被健壮的后置校验罩着 → **别在这个面上找 calldata 级的花招**。
2. 要么**去打兜底本身**，要么**换一个没有兜底的面**。
3. ⚠️ **如果兜底恰好是被排除的那一类（比如预言机 in out-of-scope）→ 这个项目对你就是死结**，早点撤。

> **算术检查**：抽取型攻击能否伤到受害方，先算这个 —— 设抵押折扣 coverage、债务 D、抵押 C：
> `remainsSolvent` 截停在 `Σ(coverage×价值) = D`。coverage ≤ 1 时 ⇒ **抵押品 ≥ D**，攻击者最多抽走 `C − D/coverage ≤ C − D` = **自有权益**。
> **结论：只要 coverage ≤ 1 且偿付检查生效，滑点/抽取型攻击永远碰不到 LP。** 不用查任何链上数值就能杀掉一整类假设。
> **唯一活路**：让 TWV **高估**真实清算价值（coverage 折扣没覆盖真实流动性滑点）—— 但那是风险参数问题，通常也不 in-scope。

## 阶段 3：生成假设 + 按证伪成本排序

### 排序规则

> **按"证伪成本"排，不按"如果成立有多爽"。**

先用最便宜的手段杀掉不成立的假设：

| 成本 | 手段 |
| --- | --- |
| 💰 最低 | **纯推理 / 算术**（如上面的 coverage 算式 —— 不用查链就杀掉整类） |
| 💰💰 | **读代码**（对照真实 ABI、追调用链） |
| 💰💰💰 | **链上只读**（`cast call` 查真实配置） |
| 💰💰💰💰 | **fork 测试**（anvil + 部署字节码） |
| 💰💰💰💰💰 | **完整 PoC + 报告** |

第二轮四条假设里，三条是靠**推理**杀掉的，零链上开销。第一轮则相反 —— 对着一个必死方向做到了最后一格。

### 通用的逻辑类攻击面类型学

| 类型 | 模式 | 去哪找 |
| --- | --- | --- |
| **Decode / Execute 分歧** | 校验解码后的视图，却转发原始字节给外部合约 —— 两个解析器 | 路由聚合器包装层、`call(abi.encodePacked(selector, data))` |
| **Fail-open on zero** | `if (x != address(0) && !valid(x)) revert` —— 零值跳过校验 | 所有 `!= address(0) &&` 的校验 |
| **不对称检查** | 一组检查路径 A 有、路径 B 没有 | 对比 `isLiquidation ? : ` 之类的分支 |
| **单向校验** | 只在"看起来亏了"时检查，"看起来赚了"直接放行 | `if (a > b) { check }` 而无 else |
| **记账 vs 真实余额** | storage 记账与 `balanceOf` 可被撬开 | 同时用两者的地方 |
| **空 catch / 吞掉失败** | `try {} catch {}`、`if (success)` 静默计 0 | 所有 `catch {}` |
| **可信角色边界** | 同名修饰器两种语义 | 见阶段 1 |

## 阶段 4：写 PoC 之前的四问闸门

**任何一问答不出来，就别写 PoC。**

1. **攻击者是谁？** 必须是不可信角色。是 owner / 白名单 → **停**。
2. **失效模式是什么？** 必须是错误的数值/状态。是 revert / 卡住 → **停**（在有 freeze 排除的项目里）。
3. **钱从谁兜里出？** 必须能论证是 LP / 协议 / 其他用户。是攻击者自己的权益 → **停**。
4. **属于哪一类 in-scope？** 必须能指名清单上的某一条。说不出来 → **停**。

---

## 阶段 5：写报告

如果四问都过了：

1. **叙事围绕损失路径，不是围绕机制。** 第一轮报告一开始写成"定价函数会 revert"，被一句"isolated pricing function, no loss-of-funds"打回。重写成"清算被阻断 → 坏账 → LP 亏损"才有说服力。
2. **给出完整调用链**，点名具体入口和行号 —— 证明"不是孤立函数"。
3. **诚实定级 + 主动暴露弱点。** triager 最反感过度包装。自己先说出减分项（自愈、条件性、无攻击者获利），反而显得分析扎实。
4. **别为了凑 in-scope 而歪曲分类。** 一眼被看穿，且伤后续提交的信誉。
5. **PoC 分层**：能打部署字节码最好，打不了就说明**为什么**打不了，并给出等价证据。

## 附 A：可复用的技术手法（工具箱）

这些是两轮里真正沉淀下来、下个项目直接能用的：

### 链上真相 > 仓库代码

| 陷阱 | 正确做法 |
| --- | --- |
| 配置项在代码里搜不到调用 ≠ 没配 | owner 可在部署后手工设置。**只能查链上状态** |
| `deployments/*.json` ≠ 链上字节码 | facet 升级后就对不上。用 `facetAddress(selector)` 从 diamond 现查 |
| vendored 的库 ≠ 部署时链接的库 | 信任根（签名者/门限）**必须从链上字节码确认** |

### 读一个没装的依赖的真实源码

```bash
# 版本号必须从 package.json / yarn.lock 抄，不是装 latest
npm pack @scope/pkg@1.2.3
tar xzf scope-pkg-1.2.3.tgz && find package -name "*.sol"
```

### anvil 上做 PoC

```bash
anvil --fork-url $RPC --fork-block-number N --port 8546
export no_proxy="127.0.0.1,localhost"   # ⚠️ 否则系统代理会拦截 localhost
```

| 目的 | 手法 |
| --- | --- |
| 把外部依赖换成受控 mock | `anvil_setCode <addr> <runtime-bytecode>` |
| 给自己授予角色 | `cast index address <me> $(cast keccak "<slot-string>")` → `anvil_setStorageAt` |
| 找角色存储在哪 | 注意可能不在目标合约上（如读的是 beacon/diamond 的存储） |
| 找阈值的真实值 | **扫一遍找翻转点**（20h OK / 26h revert → 阈值在 25h），比读字节码常量可靠 |
| 需要签名 payload 但没私钥 | **从真实交易的 calldata 尾部扒**，fork 到那个区块（时间戳才新鲜、签名者才匹配） |

### 判断"规格"vs"现实"

配置文档给的是**上界**，只有**扫历史事件**才知道现实分布。
例：Chainlink heartbeat 说 24h，实测中位数 0.5h —— 但尾部真的有 24h 的尖峰。**审计阈值类问题必须扫事件求最大值**，别信"应该不会那么久"。
