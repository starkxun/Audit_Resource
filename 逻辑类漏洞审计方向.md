# 逻辑类漏洞审计方向（通用 · 可带走）

这份文档**不绑定任何项目、也不绑定任何业务形态**。它是《赏金审计开局 checklist》**阶段 3 类型学的展开版**，同时沉淀第二轮 DeltaPrime "四条假设逐条证伪"的方法论。

三份领域文档（借贷 / 跨桥 / DEX）给的是**兜底目录**（挖哪面）；这份给的是**逻辑洞本身长什么样、怎么按证伪成本杀**（在选定的面上，具体找什么、怎么便宜地证伪）。

> 定位：兜底目录把攻击面压缩到"只剩没兜底的面"之后，**剩下的活口几乎全是逻辑洞**。这份就是那最后一公里的地图。

---

## 什么是逻辑洞

in-scope 清单里那条 **"Attacks on logic（行为 ≠ 业务描述）"** 就是它。特征：

- **不需要**预言机挂掉、不需要密码学被破、不需要 validator 合谋 —— 只需要**一条代码路径的行为偏离了它该有的意图**。
- 通常是**免费的**：无需闪电贷、无需特殊市况，一笔普通交易就触发。
- 最难的不是找到"看起来不对的代码"，而是证明它**通向资金损失且可提交**（见 §4 —— 第二轮的血泪）。

> **核心命题**：逻辑洞的技术质量和可提交性正交。一段代码"逻辑上确实错了"离"能提一个 valid bug"还差三步：不可信攻击者能触发、错误状态通向真实损失、受害方不是攻击者自己。**每条假设先过这三关，再动手写 PoC。**

## §1 逻辑洞类型学（带 grep 模式 + 代码气味）

checklist 那 7 条的展开。**每一类都给：模式 / 怎么 grep / 代码气味 / 证伪成本。**

### ① Decode / Execute 分歧
- **模式**：校验的是**解码后的视图**，转发给外部合约的却是**原始字节** —— 两个解析器，攻击者让它俩解读不一致。
- **grep**：`call(abi.encodePacked(selector, ...))`、router 聚合器包装层、`decode` 校验后又传 `rawData`
- **气味**：同一份 calldata 被 decode 一次做检查、又原样 `.call` 一次做执行。
- 💰💰 读调用链

### ② Fail-open on zero（零值短路）
- **模式**：`if (x != address(0) && !valid(x)) revert;` —— x 为零地址时**跳过整个校验**。
- **grep**：`!= address(0) &&`、`if (x != 0 &&`、`== 0 ? _ : check`
- **气味**：把"零值"当"无需检查"，但零值可能是合法且危险的输入。
- 💰 纯推理

### ③ 不对称检查（路径 A 有、路径 B 没有）
- **模式**：同一个不变量，一条分支校验了、另一条忘了。
- **grep**：`isLiquidation ? ... : ...`、`same-chain` vs `cross-chain` 分支、`if/else` 两侧行数悬殊
- **气味**：两条本该对称的路径，一条多一行 `require`，另一条没有。
- 💰💰 对比读两分支

### ④ 单向校验（只在"看起来亏了"时查）
- **模式**：`if (a > b) { check }` 而无 `else` —— 反方向直接放行。
- **grep**：`if (soldValue > boughtValue)`、`if (out < expected)` 后无对称处理
- **气味**：只防"用户占协议便宜"，不防"协议占用户便宜"或反之。
- 💰 纯推理

### ⑤ 记账 vs 真实余额
- **模式**：storage 记账值与 `balanceOf` 可被撬开（donation、fee-on-transfer、直接转账）。
- **grep**：同时用 `reserves`/`_totalDeposited` 和 `token.balanceOf(this)` 的地方
- **气味**：一个函数信记账、另一个信真实余额，两者可被外部动作拉开。
- 💰💰 读代码

### ⑥ 空 catch / 吞掉失败
- **模式**：`try {} catch {}`、`if (success)` 静默计 0，把失败当成功。
- **grep**：`catch {}`、`catch Error`、`(bool success, ) =` 后不检查 success
- **气味**：外部调用失败被吞，状态照常推进。
- 💰 纯推理 + 读

### ⑦ 可信角色边界（同名修饰器双语义）
- **模式**：`onlyOwner` 是协议 owner（可信）还是账户 owner（用户=攻击者）？
- **grep**：所有自定义 `modifier`，看它**读谁的存储**
- **气味**：修饰器名字暗示可信，实际读的是攻击者可控的存储槽。
- 💰 读代码（决定这个面挖不挖，见阶段 1）

### ⑧ 状态机时序 / 竞态（跨函数、跨链、跨区块）
- **模式**：两个操作之间存在"先证明空值、后收到写入"的窗口；或跨链两条链的高度/时间不同源。
- **grep**：`deadline` / `block.number` 跨链使用、`select` 写 transient 后 `fill` 读、cancel vs fill
- **气味**：一个不变量依赖"两件事的相对顺序"，而顺序攻击者可影响。
- 💰 纯推理（先算窗口能否同时成立）

### ⑨ 舍入 / 精度方向
- **模式**：取整方向对谁有利？先乘后除还是先除后乘？空池/首存者分支。
- **grep**：`mulDiv`、`/ ` 顺序、`totalSupply == 0 ?`
- **气味**：恒该向协议有利的取整，某条路径向用户有利。
- 💰 纯推理，但 💰💰💰💰 **PoC 必须 fork 跑出净赚**（否则被打回 theoretical）

## §2 逻辑洞藏在哪（去哪找上面这些类型）

不是逐行读，是**优先去这几个高发地带**：

| 高发地带 | 为什么 | 找什么类型 |
| --- | --- | --- |
| **分支不对称处** | 复制粘贴时漏了一侧 | ③ ④ |
| **回调 / 外部调用边界** | caller 校验往往有洞、失败被吞 | ① ⑥ ⑦ |
| **记账与真实余额并用处** | 两个真相源可被拉开 | ⑤ ⑨ |
| **状态机转移点**（open→filled→cancelled） | 转移条件、时序、重放 | ③ ⑧ |
| **同一数据被解析两次处** | 两个解析器分歧 | ① |
| **`!= 0 &&` / 三元短路** | 零值/边界跳过校验 | ② |
| **权限修饰器定义处** | 名字骗人 | ⑦ |

## §3 证伪纪律（第二轮 DeltaPrime 的方法论）⭐

这是本文最值钱的部分。**逻辑洞最容易翻车在"找到了就一头扎进写 PoC"。** 正确姿势：

### 排序规则

> **按"证伪成本"排，不按"如果成立有多爽"。** 先用最便宜的手段杀掉不成立的假设。

| 成本 | 手段 | 能杀掉什么 |
| --- | --- | --- |
| 💰 最低 | **纯推理 / 算术** | 单向校验、零值短路、时序窗口能否同时成立 |
| 💰💰 | **读代码**（对照真实 ABI、追调用链） | decode/execute 分歧、不对称检查、记账偏移 |
| 💰💰💰 | **链上只读**（`cast call` 查真实配置/余额） | "这个 impl 真部署了吗""这个 token 有价值吗" |
| 💰💰💰💰 | **fork 测试**（anvil + 部署字节码） | 舍入型净赚、复杂多步交互 |
| 💰💰💰💰💰 | **完整 PoC + 报告** | 只留给活到最后的那一条 |

### 第二轮案例（四条假设，三条纯推理杀掉，零链上开销）

> 方向：ParaSwap calldata 逻辑洞。四条假设逐条证伪：
> - 三条靠**纯推理**杀掉（撞可信角色 / 撞兜底机制 / 无攻击者获利）
> - 一条待验且预期不可利用
> **教训**：第一轮相反 —— 对着一个必死方向做到了完整 PoC 才发现方向不可提。**证伪成本排序能把 90% 的死方向在第一格就杀掉。**

### 本轮 HyperBridge 印证

> 信任根层三条假设（MMR 峰值、门限计数、非成员证明+空值键）全用 💰~💰💰 杀掉，没进 fork。**最深的那条（非成员证明）也只追到底层存储值类型就证伪，没写一行 PoC。**

## §4 四问闸门（逻辑洞专用，最容易死在这）

逻辑洞"技术上真"但提不成的三种典型死法，**对应四问逐个防**：

| 问 | 逻辑洞的典型翻车 |
| --- | --- |
| **1. 攻击者是谁？** | ❌ 触发需要 owner / 白名单 / 可信 relayer / 可信角色 → **第二轮全军覆没在这** |
| **2. 失效模式？** | ❌ 只是 revert / 卡住 → 撞 temporary-freeze 排除（第一轮死这） |
| **3. 钱从谁兜里出？** | ❌ 受害方是攻击者自己（本金/滑点/无常损失）；或自愈；或 solver 自承风险无 LP 损失 → **griefing，定级弱** |
| **4. 哪类 in-scope？** | ❌ 说得出"逻辑不对"，说不出"损失路径" → triager 一句 "isolated function, no loss of funds" 打回 |

> **写 PoC 前，四问必须全绿。** 任何一问红灯 → 这条假设就地放弃，无论代码看起来多不对。

## §5 写报告（逻辑洞叙事）

| 要点 | 说明 |
| --- | --- |
| **叙事绕损失路径，不绕机制** | 别写"X 函数逻辑错了"，写"攻击者调 A → 触发 X 的错误分支 → 抽走 B → 受害方 C 亏损"。第一轮"定价函数会 revert"被打回，重写成"清算阻断 → 坏账 → LP 亏"才有说服力 |
| **给完整调用链 + 行号** | 证明"不是孤立函数"，点名入口 |
| **诚实定级 + 主动暴露减分项** | 自愈、条件性、无攻击者获利 —— 自己先说，显得分析扎实 |
| **别为凑 in-scope 歪曲分类** | 一眼被看穿，伤后续信誉 |
| **PoC 分层** | 能打部署字节码最好；打不了要说清**为什么**，给等价证据 |
